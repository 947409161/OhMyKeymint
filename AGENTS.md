# Guide

## Build & dev setup

- Nightly Rust is required (`#![feature(once_cell_try)]`).
- Target: `aarch64-linux-android` (primary). `x86_64-linux-android` is also supported.
- Android NDK is required. Generate `.cargo/config.toml`:
  `python scripts/setup_cargo_config.py --ndk-root <path>`
- BoringSSL is an external C dependency built separately. Set `BORINGSSL_BUILD_DIR` env var
  to the CMake build output directory (e.g. `~/.cargo/boringssl/build`).
- AIDL code is generated by `build.rs` (root) and `injector/build.rs` into `OUT_DIR/aidl.rs`.
  Protobuf (`proto/storage.proto`) is compiled into `src/proto/` by `build.rs`.
- Build: `cargo build --target aarch64-linux-android`
- Clippy: `cargo clippy --target aarch64-linux-android --all-targets`
- Format: `cargo fmt --all`
- Tests only run on a real Android device. See deploy steps below.

## Architecture

- Two binaries:
  - `keymint` (root `src/main.rs`) — the keymint HAL daemon implementing `IKeyMintDevice` etc.
  - `inject` (workspace member `injector/`) — ptrace-based injector that patches
    `android.hardware.security.keymint` and `android.system.keystore2` service threads.
- Workspace members: `boringssl`, `common`, `derive`, `ta`, `tests`, `wire`, `watchdog`, `injector`.
- `src/keymaster/` — core keystore logic (service, crypto, DB, enforcements, super key).
- `src/keymint/` — KeyMint HAL wrapper backends.
- `hal/` — HAL service implementation (AOSP-derived, CBOR-serialized channel to TA).
- `refs/` — reference material only, not compiled. Ignore for build purposes.
- `template/` — Magisk/KSU module shell structure. `build.py` assembles these + binaries into a zip.
- Code style: 4-space indent, LF line endings (`.editorconfig`).

## Device deployment

- Hot update (no reboot): `python scripts/deploy_hot_update.py --serial <serial> --abi arm64-v8a --restart all`
  Builds, deploys to `/data/adb/omk/`, verifies hashes, hot-restarts both daemons.
- Full module zip: `python build.py --release --abi arm64-v8a`
- Restart daemons manually on device:
  `touch /data/adb/omk/restart.keymint` (or `restart.injector`, `restart.all`)

## Config

- Runtime config: `/data/misc/keystore/omk/config.toml` (keymint) and `injector.toml` (injector).
- OMK rewrites `config.toml` after a successful result, filling in generated crypto seeds.
  An invalid config is backed up to `config.toml.bak` with the parse error appended.
- `[trust].security_patch` changes are hot-applied without restart.
  `[trust].vb_key` / `vb_hash` changes from `"random"` to `"auto"` require a full reboot.

## Domain invariants (do not break)

- Scoop-routed applications: OMK is the authoritative business path. If OMK is reachable and
  returns a business error, return that OMK error as-is. Do not swallow, downgrade, or fallback
  to a system success reply. Only OMK-unavailable failures (missing backend, connect failure,
  `DeadObject`) may preserve the system reply.
- Auth-bound APP key storage:
  - injector must not patch auth-bound APP `generateKey` request flags.
  - `KeystoreSecurityLevel::store_new_key_descriptor` must pass the original flags to
    `SUPER_KEY.handle_super_encryption_on_key_init`; do not add CE-availability probes or a retry
    that ORs in `KEY_FLAG_AUTH_BOUND_WITHOUT_CRYPTOGRAPHIC_LSKF_BINDING`.
  - `Enforcements::super_encryption_required` is the only place where that flag disables
    cryptographic LSKF binding.
  - If `SuperEncryptionType::CredentialEncrypted` has no unlocked CredentialEncrypted super key,
    keep the current `LOCKED` / `UNINITIALIZED` result. Do not convert that state into successful
    key storage based on `sys.user.<id>.ce_available` or similar Android CE-availability signals.
- Config changes must be reflected in `README.md` and the `template/` directory.

## Tooling conventions

- Use `rg` for searching. Keep edits ASCII unless the file already uses non-ASCII.
- Keep it pure Rust unless you get explicit approval.
- Run tests on a real Android device after changes. Do not rely on host-only test artifacts.
- Make clippy happy. `#[allow(clippy::*)]` requires justification.
- Check for existing implementations before writing new ones. Reuse, don't duplicate.
