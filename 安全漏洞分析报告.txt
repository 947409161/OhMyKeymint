Oh My Keymint 安全漏洞分析报告
================================
分析日期: 2026-06-13
分析范围: Injector 模块、OMK 主服务、配置管理、平台伪装

================================================================================
                           一、高危漏洞 (High)
================================================================================

[漏洞 1] Injector ptrace 注入竞态条件
--------------------------------------------------------------------------------
位置: injector/src/inject.rs:283-331
类型: 竞态条件 (Race Condition)

详细描述:
  inject_library() 函数使用 ptrace attach 到 keystore2 进程后，
  在 get_regs() 和 set_regs() 之间存在时间窗口。
  如果 keystore2 进程在该窗口内执行信号处理或线程调度，
  可能导致寄存器状态不一致。

触发条件:
  - 高负载系统
  - keystore2 进程多线程环境
  - 信号注入时机不当

潜在影响:
  - 注入失败
  - 目标进程崩溃
  - 远程内存损坏
  - 潜在的代码执行（如果 regs 状态被恶意利用）

严重程度: 高

修复建议:
  - 使用 PTRACE_SEIZE 而非 PTRACE_ATTACH
  - 注入期间阻塞所有信号
  - 添加更多的错误检查和回滚机制


[漏洞 2] Parcel 反序列化数据验证不足
--------------------------------------------------------------------------------
位置: injector/src/parcel.rs:998-1020, 1061-1083
类型: 拒绝服务 / 潜在内存损坏

详细描述:
  parse_typed_request() 和 read_reply_binder_carrier() 函数
  在反序列化 Parcel 数据时未进行充分的边界验证。
  虽然 parcel_from_ipc_parts() 在空数据时使用 dangling pointer，
  但后续的 read() 操作可能访问无效内存。

触发条件:
  - 发送恶意构造的 Binder transaction 数据
  - 截断的或格式错误的 Parcel payload

潜在影响:
  - 进程崩溃 (DoS)
  - 信息泄露（通过越界读取）
  - 在某些情况下可能实现代码执行

严重程度: 高

修复建议:
  - 在反序列化前验证数据大小
  - 添加显式的边界检查
  - 对 read_reply_binder_carrier() 中的 data_size 验证


[漏洞 3] 配置文件权限管理缺陷
--------------------------------------------------------------------------------
位置: src/config.rs:111-128, src/main.rs:74-89
类型: 敏感数据暴露

详细描述:
  persist_config_file() 在写入配置文件后设置权限为 0o660，
  但在首次创建或文件不存在时没有立即设置权限。
  如果在权限设置前文件被其他进程读取，可能导致种子密钥泄露。

触发条件:
  - 配置文件首次创建
  - 文件权限恢复出厂设置后
  - 多用户环境

潜在影响:
  - 加密种子密钥泄露 (root_kek_seed, kak_seed, shared_secret_seed)
  - 设备绑定密钥泄露
  - 完整的密钥恢复攻击

严重程度: 高

修复建议:
  - 使用原子性文件写入（写临时文件后 rename）
  - 在写入完成后立即设置权限
  - 考虑使用 Android KeyStore 存储敏感配置


[漏洞 4] 超级密钥内存安全问题
--------------------------------------------------------------------------------
位置: src/keymaster/super_key.rs, src/keymaster/gc.rs:132-140
类型: 密钥内存泄露

详细描述:
  SuperKeyManager 使用 ZVec 存储加密密钥，但：
  1. unwrap_key_if_required() 可能将解密后的明文密钥保留在内存中
  2. gc.rs 中删除密钥时，密钥数据可能未被安全擦除
  3. 在堆内存中以 Vec<u8> 形式传递密钥，存在被泄露风险

触发条件:
  - 内存转储攻击 (memory dump)
  - 冷启动攻击 (cold boot attack)
  - 调试器附加

潜在影响:
  - 加密密钥明文泄露
  - 所有受保护密钥的完全恢复
  - 设备绑定失效

严重程度: 高

修复建议:
  - 使用 secure_zero() 替代 Vec<u8> 进行密钥存储
  - 在 gc 处理后显式覆写内存
  - 考虑使用 mlock() 防止密钥交换到磁盘

                           二、中危漏洞 (Medium)

[漏洞 5] ioctl Hook 绕过风险
--------------------------------------------------------------------------------
位置: injector/src/hook/mod.rs:18-20, injector/src/hook/intercept.rs:37-77
类型: 安全控制绕过

详细描述:
  new_ioctl() 函数直接调用原始 ioctl 而不验证请求。
  如果 hook 安装失败或 OLD_IOCTL 未正确初始化，
  可能导致 binder 通信异常或安全问题。

触发条件:
  - hook 安装不完整
  - 多实例 ioctl 调用冲突
  - OLD_IOCTL 指针损坏

潜在影响:
  - Binder 通信中断
  - 密钥操作失败
  - 安全检查被绕过

严重程度: 中

修复建议:
  - 添加 OLD_IOCTL 初始化验证
  - 实现请求完整性检查
  - 添加 hook 状态监控


[漏洞 6] 包名过滤可绕过
--------------------------------------------------------------------------------
位置: injector/src/filter.rs:26-100, injector/src/identify.rs:153-168
类型: 权限绕过

详细描述:
  1. allow_unknown_package = true 时，所有未知包都会被放行
  2. android 包默认被 block_android_package 拦截
  3. scoop 白名单验证依赖包名解析，可能被伪造

触发条件:
  - 配置 allow_unknown_package = true
  - 恶意应用伪造包名
  - 包名解析失败

潜在影响:
  - 未授权应用绕过 OMK 拦截
  - 密钥被恶意应用访问
  - 认证信息被窃取

严重程度: 中

修复建议:
  - 禁用 allow_unknown_package
  - 添加 APK 签名验证
  - 实现更强的包名验证机制


[漏洞 7] vbmeta 伪造完整性问题
--------------------------------------------------------------------------------
位置: src/plat/vbmeta.rs:52-80, src/plat/attestation.rs:156-234
类型: 信任链伪造

详细描述:
  OMK 从系统属性读取 vbmeta 信息并可配置为 "auto" 模式。
  如果系统属性被恶意修改（如通过 resetprop），可能导致：
  1. verified_boot_state 被伪造
  2. vb_hash/vb_key 被篡改
  3. attestation 信息不一致

触发条件:
  - resetprop 被恶意利用
  - SELinux 策略被降级
  - 系统分区被修改

潜在影响:
  - 设备认证被欺骗
  - 安全级别检测失效
  - 远程 attestation 欺骗

严重程度: 中

修复建议:
  - 验证 vbmeta 签名
  - 缓存关键属性值
  - 添加启动时完整性检查


[漏洞 8] ID 旋转时间戳文件安全
--------------------------------------------------------------------------------
位置: src/keymaster/id_rotation.rs:37-62
类型: 时间戳操纵

详细描述:
  had_factory_reset_since_id_rotation() 依赖文件修改时间作为
  factory reset 检测依据。文件时间可通过 touch 命令修改。

触发条件:
  - adb 权限
  - root 权限
  - SELinux 策略被降级

潜在影响:
  - ID 旋转逻辑被绕过
  - 密钥可能在新设备上被重用
  - 设备绑定失效

严重程度: 中

修复建议:
  - 使用 trusted timestamp (如 RPMB)
  - 结合 eMMC/GPT 特性
  - 验证文件时间与系统启动时间一致性


[漏洞 9] 请求转发逻辑安全问题
--------------------------------------------------------------------------------
位置: injector/src/forward.rs, injector/src/hook/rewrite.rs:728-1330
类型: 请求路由错误

详细描述:
  handle_br_transaction() 中的路由决策基于包名和配置，
  但存在以下问题：
  1. precompute_reply() 可能计算错误的回复
  2. on_reply_failure() 在失败时可能保留系统回复
  3. pending_preserves_system_on_rewrite_failure() 逻辑复杂

触发条件:
  - 高并发请求
  - 快速重启场景
  - OMK 服务无响应

潜在影响:
  - 白名单应用收到系统 keystore 回复
  - 密钥泄露到系统 keystore
  - 认证令牌被错误路由

严重程度: 中

修复建议:
  - 简化路由决策逻辑
  - 添加路由一致性验证
  - 实现超时和回退机制


[漏洞 10] Tracker 模块状态管理
--------------------------------------------------------------------------------
位置: injector/src/tracker.rs, injector/src/hook/rewrite.rs:220-240
类型: 状态一致性

详细描述:
  KeyDescriptor 路由通过 grant 追踪，但：
  1. tracker 状态在进程重启后丢失
  2. 状态可能与实际路由不一致
  3. 无锁保护的并发访问

触发条件:
  - 并发密钥操作
  - 快速 grant/ungrant 序列
  - injector 进程重启

潜在影响:
  - 密钥路由到错误后端
  - 认证状态不一致
  - 潜在的密钥访问控制绕过

严重程度: 中

修复建议:
  - 实现持久化 tracker 状态
  - 添加并发访问同步机制
  - 实现状态一致性验证

[漏洞 11] 软件实现密码学
--------------------------------------------------------------------------------
位置: src/keymint/soft.rs, src/keymint/sdd.rs
类型: 密码学实现风险

详细描述:
  软件实现的密钥派生和 SDD 存储使用 Rust 默认随机数生成器。
  虽然使用了 BoringRng，但在某些平台上可能不够安全。

触发条件:
  - 熵不足的环境
  - 虚拟化环境
  - 重复启动的容器环境

潜在影响:
  - 密钥可预测性增加
  - 随机数重复使用

严重程度: 低

修复建议:
  - 使用 getrandom() 或 android_random 扩展
  - 添加启动时熵收集
  - 实现密码学安全的 RNG


[漏洞 12] 日志泄露敏感信息
--------------------------------------------------------------------------------
位置: 多处 debug/info 日志语句
类型: 信息泄露

详细描述:
  代码中包含多处日志语句可能输出敏感信息：
  - binder transaction 详情
  - 密钥描述符
  - 包名和 UID

触发条件:
  - 日志级别设置为 debug
  - logcat 输出被捕获
  - 文件日志被读取

潜在影响:
  - 密钥路径泄露
  - 应用包名暴露
  - 调试信息泄露

严重程度: 低

修复建议:
  - 生产环境禁用 debug 日志
  - 敏感字段脱敏
  - 日志访问控制


[漏洞 13] resetprop helper 权限
--------------------------------------------------------------------------------
位置: src/plat/resetprop.rs, src/main.rs:160
类型: 权限提升

详细描述:
  bootstrap_privileged_helper() 使用 root 权限创建 helper 进程。
  如果 helper 二进制文件权限配置不当，可能被利用。

触发条件:
  - SELinux 策略被降级
  - 文件权限配置错误
  - 系统漏洞导致提权

潜在影响:
  - 系统属性被恶意修改
  - vbmeta 信息伪造
  - 设备认证欺骗

严重程度: 低

修复建议:
  - 验证 helper 二进制签名
  - 限制 helper 能力
  - 添加 SELinux 策略限制


[漏洞 14] GC 异步操作竞态
--------------------------------------------------------------------------------
位置: src/keymaster/gc.rs:79-87, 145-165
类型: 竞态条件

详细描述:
  notify_gc() 使用 compare_exchange 防止重复通知，
  但 step() 函数中的递归调用可能导致：
  1. 多个 gc 实例并发运行
  2. 密钥被多次删除
  3. 数据库状态不一致

触发条件:
  - 快速连续的 notify_gc 调用
  - 高负载系统
  - 数据库锁定超时

潜在影响:
  - 数据库锁定冲突
  - 密钥删除失败
  - 资源泄漏

严重程度: 低

修复建议:
  - 添加互斥锁
  - 实现批处理优化
  - 添加重试限制

================================================================================
                           四、设计层面问题
================================================================================

[问题 1] 整体架构安全性依赖于 SELinux
--------------------------------------------------------------------------------
描述:
  OMK 大量依赖 SELinux 策略来保护：
  - /data/misc/keystore/omk/ 目录访问
  - binder 服务访问控制
  - resetprop helper 权限

风险:
  如果 SELinux 被禁用或策略被降级，所有安全控制失效。


[问题 2] 密钥管理缺乏硬件绑定
--------------------------------------------------------------------------------
描述:
  OMK 在 Injector 模式下将密钥存储在文件中，
  密钥由配置文件中的种子派生，缺少 TEE/Strongbox 绑定。

风险:
  提取 /data/misc/keystore/omk/ 目录后可能恢复密钥。


[问题 3] 平台伪造的一致性问题
--------------------------------------------------------------------------------
描述:
  OMK 伪造设备信息（品牌、型号、vbmeta 等），
  但伪造值与实际硬件可能存在不一致。

风险:
  高级检测工具可能通过多源信息交叉验证发现异常。

================================================================================
                           五、漏洞利用场景
================================================================================

场景 1: 从检测工具角度绕过 OMK
--------------------------------------------------------------------------------
攻击者使用 root 权限直接读取 /data/misc/keystore/omk/ 目录，
提取种子密钥和 keybox.xml，实现完整的密钥恢复。

条件: root 权限
影响: 密钥恢复，设备绑定失效


场景 2: 伪造的 attestation
--------------------------------------------------------------------------------
通过修改 resetprop 修改系统属性，欺骗 OMK 的 vbmeta 解析逻辑，
实现伪造的 verified boot state。

条件: root + resetprop 能力
影响: 设备认证欺骗


场景 3: injector hook 绕过
--------------------------------------------------------------------------------
检测 keystore2 进程的内存映射，发现被注入的 .so，
通过 unlink 或 mmap 覆写移除 hook，恢复原生行为。

条件: root + ptrace 能力
影响: OMK 失效


场景 4: Parcel 畸形数据攻击
--------------------------------------------------------------------------------
发送恶意构造的 Binder transaction 数据到 OMK 服务，
触发解析错误或内存损坏。

条件: 可与 keystore2 通信
影响: 服务崩溃或代码执行

================================================================================
                           六、安全建议总结
================================================================================

高优先级:
1. 修复 Parcel 反序列化边界验证
2. 改进配置文件原子性写入
3. 实现超级密钥安全内存管理
4. 添加 ptrace 注入完整性检查

中优先级:
5. 禁用 allow_unknown_package 默认配置
6. 实现 vbmeta 签名验证
7. 改进 tracker 状态持久化
8. 简化请求转发逻辑

低优先级:
9. 日志脱敏
10. 改进 GC 并发安全
11. 随机数生成器审计
